## 주요 기술 결정 및 맥락

- **API 적재를 Spring Boot로 구현**: CSV와 달리 API 데이터는 JSON 파싱 + 엔티티 변환이 필요 → JPA 활용이 자연스러움
- **TransactionTemplate 사용**: 2.5만 건을 하나의 트랜잭션으로 처리하면 메모리 부족/타임아웃 위험 → 페이지(100건) 단위로 Commit 분리
- **비동기 실행 (CompletableFuture)**: API 250페이지 순회에 수 분 소요 → 브라우저 응답은 즉시 반환하고 백그라운드 처리
- **@JsonIgnoreProperties(ignoreUnknown = true)**: 공공데이터 API 응답에 예상 외 필드가 있어도 파싱 실패 방지
- **BGE-M3 (Ollama) 선택**: OpenAI/Google 임베딩 API 비용 대신 로컬 Ollama 사용. 1024차원, 무료, 한국어 지원

### 4. knowledge_embeddings 적재 (Python RAG 구축)

**목표**: `drug_easy_info` 테이블의 약물 정보(상호작용, 주의사항 등)를 벡터화하여 RAG 지식 베이스 구축.

### 생성·수정 파일

| 파일 | 역할 |
|------|------|
| `ai-server/app/core/database.py` | `psycopg` DB 연결, `fetch_drug_easy_info()` — 임베딩 대상 데이터 조회 |
| `ai-server/app/services/ollama_embedding_service.py` | Ollama 로컬 BGE-M3로 임베딩 생성 (`get_embedding`) |

| `ai-server/scripts/embed_drug_info.py` | 배치 스크립트 — drug_easy_info 조회 → 임베딩 → DB 적재 |

### 의존성 추가 (`requirements.txt`)

- `ollama` — 로컬 Ollama API 호출
- `tqdm` — 진행률 표시

### 임베딩 모델 결정

| 후보 | 차원 | 비용 | 최종 |
|------|------|------|------|
| OpenAI text-embedding-3-small | 1536 | 유료 | × |
| Google text-embedding-004 | 768 | 무료(쿼터) | × |
| **BGE-M3 (Ollama)** | **1024** | **무료** | ✓ |

- Ollama에 BGE-M3 설치 후 로컬에서 실행, API 비용 없음.

### knowledge_embeddings 컬럼 활용

| 컬럼 | 역할 |
|------|------|
| `content` | RAG 시 LLM에 전달하는 컨텍스트 (약품명, 효능, 주의사항, 상호작용 등) |
| `embedding` | 코사인 유사도 검색용 벡터 (BGE-M3, 1024차원) |
| `metadata` | `item_seq`, `item_name`, `entp_name` — 검색 결과 출처 표시 |
| `source`, `category` | 출처·용도 필터링 (`drug_easy_info`, `drug_interaction`) |

### RAG 흐름 (상호작용 분석)

```
[입력] 약물 A + 영양제 B
  → 쿼리 임베딩 (예: "약물 A 성분 X와 영양제 B 성분 Y의 상호작용")
  → pgvector: embedding <=> 쿼리벡터 ORDER BY LIMIT k
  → 상위 k개 content를 LLM 컨텍스트로 전달
  → LLM 출력: SAFE / CAUTION / WARNING / SYNERGY 등급
```

## 데이터 분석 및 방향 재설정

### 1. 낱알식별·성분 매칭 현황

| 항목 | 건수 |
|------|------|
| 매칭 (성분 정보 있음) | 21,056 |
| 비매칭 (성분 정보 없음) | 3,688 |
| **매칭률** | **약 85%** |

**결정**: 비매칭 데이터 **유지**. 검색 품질 보장, 선택 시 On-demand 보강 또는 "정보 없음" 안내.

---

### 2. 상세 정보(효능·주의·상호작용) 한계

- **drug_easy_info (e약은요)**: 약 4,700건 — **API 전체 데이터 이미 적재됨**
- e약은요는 "공급실적이 있는 일반의약품" 위주 → **전문의약품 상세 정보 부족**
- drugs 2.5만 건 중 drug_easy_info 매칭률 **약 20%**
- 일반의약품은 포장지에 설명 있음. **검색이 필요한 전문의약품** 정보가 부족함

---

### 3. 약학정보원 수준 데이터 확보 불가

- 공공데이터(DUR, 병용금기)는 **약물-약물**만. **약물-영양제** 정형 데이터 없음
- 추가 가능 소스: e약은요 intrcQesitm(이미 보유), supplements IFTKN_ATNT_MATR_CN, 허가 상세 EE_DOC_DATA(응답 확인 필요), 해외 DB(성분 매칭), 문헌

---

### 4. LLM 단독 의존 불가

- 데이터 없이 LLM이 "이 약은 ~입니다" 답변 → **환각 위험**
- 의료·건강 맥락에서 신뢰 불가
- **데이터 확보 우선**, LLM은 데이터 기반 보조

---

### 5. 방향: 국내 의약품 데이터 수집 → RAG 기반 답변

- 공공데이터·e약은요 등 수집 → 임베딩 → RAG
- LLM이 **해당 데이터만** 참조해 답변 (환각 최소화)

---

### 6. 흐름 설계

```
1차: 낱알식별 (drug_identification) → drug 선택
2차: drug_easy_info 있으면 → DB에서 바로 반환
     없으면 or 상호작용 질문 → RAG (한 번에, 약품명+성분+질문 포함)
```

- RAG 검색 시 **category 메타데이터 필터링** (질문 유형별로 drug_efficacy, drug_interaction 등 구분)
