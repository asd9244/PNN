# 2026-03-21 PnN 프로젝트 개발 일지 (Devlog)

이번 일지는 **React Native 앱(`pnn-app`)** 위주로, 홈·영양제 추천·약품 상세·상호작용 비교 결과 화면의 UX 개선과 버그 수정 내용을 날짜 단위로 묶어 정리합니다. 백엔드·AI 서버 로직의 대규모 변경은 포함하지 않으며, 앱에서 사용자가 보는 흐름과 스토어·네비게이션 연동에 초점을 둡니다.

## 2. 영양제 추천 결과 화면 — UI 재구성 및 「제품 검색 링크」 화면 신설

### 2.1 이모지·외부 쇼핑 버튼 정리 (`RecommendationResultScreen.tsx`)
- 추천 사유·주의사항 라벨에 붙어 있던 **💡**, **⚠️** 등 장식 이모지를 제거하고, 텍스트만으로 「추천 사유」「주의사항」으로 표시하도록 정리했습니다.
- 카드 하단에 있던 **Coupang / iHerb** 개별 버튼과 `Linking.openURL` 직접 호출 로직을 **삭제**했습니다.
- 대신 **한 줄짜리 CTA 버튼** 「이 성분이 포함된 제품 찾아보기」만 두고, 탭 시 **별도 스택 화면**으로 이동하도록 변경했습니다.

### 2.2 링크 전용 화면 (`ProductSearchLinksScreen.tsx`) 및 유틸
- **목적:** 쿠팡·iHerb 검색 URL 규칙을 한곳에서 관리하고, 사용자가 외부 쇼핑몰로 나가기 전에 **성분별로 링크를 모아서** 볼 수 있게 함.
- **`utils/productSearchLinks.ts`**
  - `getProductSearchLinks(nameKr, nameEn)` → `{ label: 'Coupang' \| 'iHerb', url }[]` 반환.
  - 쿠팡: `Platform.OS === 'web'` 이면 데스크톱 검색 URL, 아니면 모바일 검색 URL.
  - iHerb: `https://kr.iherb.com/search?kw=...` (기존 앱 내 하드코딩과 동일한 검색어 규칙 유지).
- **`ProductSearchLinksScreen.tsx`**
  - 헤더 제목: 「제품 검색 링크」.
  - 라우트 파라미터: 추천된 **모든 성분**을 한 번에 받도록 `{ nutrients: { nameKr, nameEn }[] }` 형태로 설계.
  - 각 성분마다 한글명·영문명 제목 아래에 Coupang / iHerb 행을 나열하고, 행 탭 시 `Linking.openURL`.

### 2.3 CTA 위치 — 스크롤 최하단 단일 버튼
- 초기에는 **성분 카드마다** 「이 성분이 포함된 제품 찾아보기」가 붙어 있었으나, 요청에 따라 **카드 내부 버튼을 제거**했습니다.
- **유의사항(면책) 블록 아래**, 스크롤 맨 끝에 **버튼을 하나만** 배치했습니다.
- 탭 시 `result.recommendedNutrients` 전체를 `nutrients` 배열로 넘겨, 링크 화면에서 **추천된 모든 성분**의 검색 링크를 한 화면에 표시합니다.

### 2.4 네비게이션 (`AppNavigator.tsx`)
- `RootStackParamList`에 `ProductSearchLinks: { nutrients: { nameKr: string; nameEn: string }[] }` 추가.
- `Stack.Screen` 등록 및 스타일은 `ProductSearchLinksScreenStyles.ts`, `styles/index.ts` export로 연결.

### 2.5 스타일 (`RecommendationResultScreenStyles.ts` 등)
- 쿠팡/아이허브 전용 색상·이중 버튼 스타일을 제거하고, 보라색 단일 CTA(`productSearchButton` / `productSearchButtonText`)와 하단 섹션(`bottomProductSearchSection`)만 유지.

## 4. 성분 데이터가 없는 약 — 요약 문구·「약 추가」 동작 정리

### 4.1 요약 리스트에서 「성분 정보 없음」 제거
- 이전에는 `ingredientNamesEng`가 비어 있을 때 bullet 대신 **「성분 정보 없음」** 문구를 보여 주었습니다.
- 요청에 따라 해당 문구를 **표시하지 않도록** 변경했습니다. 성분이 있을 때만 `• 영문 성분` 목록을 렌더링합니다.

### 4.2 약 추가 시 안내 → **추가 자체를 막음** (`DrugDetailScreen.tsx`)
- 한때는 성분이 없어도 Alert 후 목록에 추가되도록 했으나, **이름만 있는 약은 비교·추천 결과에 기여하지 못한다**는 이유로 정책을 바꿨습니다.
- `ingredientNamesEng`를 필터한 결과가 **빈 배열**이면:
  - 스토어(`addInteractionDrug` / `addRecommendationDrug`)에 **넣지 않음**.
  - `InteractionCheck` / `Recommendation`으로 **pop·navigate 하지 않음**.
  - 사용자에게만 안내(네이티브 `Alert`, 웹 `window.alert`)로 「성분 정보가 있는 약품만 목록에 추가할 수 있다」는 뜻을 전달.

### 4.3 「약 추가」 버튼이 반응 없던 문제 — 원인과 수정
- **원인 추정**
  1. **이중 헤더:** `DrugDetail`에 스택 기본 헤더(`headerShown: true`)와 화면 내부 커스텀 `Header`가 동시에 있어, 웹/일부 환경에서 터치 레이어가 꼬일 수 있음.
  2. **헤더 레이아웃:** 제목 `Text`에 `flex: 1`이 붙어 긴 약품명일 때 오른쪽 「약 추가」 영역과 겹쳐 **터치가 제목 쪽으로만 잡히는** 현상 가능.
  3. **웹에서 `Alert.alert`:** `react-native-web` 환경에 따라 다이얼로그가 기대대로 뜨지 않을 수 있어, 성분 없음 분기에서 **아무 반응 없음**처럼 보일 수 있음(이후 정책상 성분 없으면 Alert만이 아니라 추가도 막음).

- **조치**
  - `AppNavigator`에서 `DrugDetail`에 **`headerShown: false`** — 상세는 커스텀 `Header`만 사용.
  - `Header.tsx`: 제목을 `titleWrap`으로 감싸고 `numberOfLines={1}`, `ellipsizeMode="tail"`, 오른쪽 영역에 `zIndex` / `elevation` 부여.
  - `DrugDetailScreen`: 「약 추가」에 `hitSlop`, `activeOpacity` 보강.
  - 성분 없음 분기는 웹에서 `window.alert`로 안내(추가는 하지 않음).

## 5. 상호작용 비교 결과 — 「성분 없음」 전용 안내 UI 제거 (`InteractionResultScreen.tsx`)

### 배경
- 앱 단계에서 이미 **성분 없는 약은 목록에 넣지 않도록** 바꿨기 때문에, 결과 화면에서 **`nutrient === '알 수 없음'`** 인 항목(백엔드·AI 쪽 성분 부족 fallback)을 **별도로 안내하던 UI**는 중복·혼란 요소가 됩니다.

### 변경 내용
- 응답의 `interactions` 중 **`nutrient === '알 수 없음'`** 인 항목은 **화면에 그리지 않도록** 필터링 (`visibleInteractions`).
- 그룹핑·아코디언은 `visibleInteractions`만 사용.
- 상단 「분석 결과 요약」에서
  - 이전: 실제 상호작용 / **성분 데이터 부족만 있는 경우** / 그 외 로 3분기.
  - 이후: **`visibleInteractions.length > 0`** 이면 「상호작용이 감지된 항목이 있습니다.」, 아니면 「특이한 상호작용이 발견되지 않았습니다.」로 단순화.
- 카드 내부의 **「데이터 부족」** 라벨, 성분 쌍 대신 설명만 크게 보여 주던 레이아웃, 「안내 사항」 분기 등을 제거하고, 남은 항목은 모두 **영양 성분 × 처방약 성분** + 설명 + 안전 복약 가이드 형태로 통일.

> 참고: 백엔드(`InteractionCheckService`, Python `interaction.py`)는 여전히 성분 없을 때 fallback 객체를 만들 수 있으나, 앱에서는 위 필터로 **표시하지 않음**. 서버 쪽 분기를 완전히 제거할지는 추후 API 정리 시 검토 가능.

## 7. 후속으로 열릴 수 있는 과제 (메모)

- 백엔드에서 더 이상 `알 수 없음` fallback을 내려주지 않도록 정리할 경우, **프론트 필터**는 이중 안전망으로 유지할지 제거할지 결정.
- `DrugDetail`의 `headerShown: false` 이후 노치·상태바 영역은 `SafeAreaView`로 커버 중이나, 기기별로 상단 여백을 한 번 더 점검하면 좋음.
