## 4. 논의 및 설계 결정 사항 (검색·낱알식별 API)

### 4.1 약품 상세 검색 API (`/api/drugs/search/detail`)

- **목적**: 이름, 제조사, 성분명 기반 검색
- **입력**: `itemName`, `entpName`, `ingredient` (최소 1개 필수, AND 결합)
- **응답**: `drugId`, `itemName`, `entpName`, `itemImageUrl` (리스트 뷰에 필요한 최소 정보만 반환)
- **구현**: 성분명 검색 시 `drug_ingredients` 테이블 조인, QueryDSL 사용 동적 쿼리 처리

### 4.2 약품 낱알 식별 검색 API (`/api/drugs/search/pillIdentifier`)

- **목적**: 알약의 물리적 특성 기반 검색
- **입력**: `printFront`, `printBack`, `markCode`, `drugShape`, `color`, `line`, `formulation` (최소 1개 필수)
- **결합 규칙**:
  - 다른 속성 간에는 AND 결합
  - 앞/뒤 구분이 모호한 속성(각인, 색상, 분할선, 마크)은 내부적으로 **OR 결합** (예: `colorFront LIKE '%하양%' OR colorBack LIKE '%하양%'`)
- **응답**: 상세 검색 API와 동일한 DTO 재사용
- **특이사항**:
  - 식별 마크는 이미지 URL(`mark_img`) 대신 고유 코드(`mark_code`)를 사용하여 검색 성능 및 정확도 향상.
  - 제형(`formulation`)은 현재 DB의 `form_code_name` 값이 너무 파편화되어 있어, 추후 수동 매핑을 통해 3~4개 카테고리로 정규화한 뒤 적용 예정.
