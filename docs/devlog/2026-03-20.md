## 1. 프론트엔드 UI/UX 개선 작업

### 안전 영역(SafeArea) 및 하단 겹침 문제 해결
- **문제점:** 앱 하단의 `ActionButtons` 및 고정 요소들이 일부 기기에서 안드로이드/iOS 시스템 하단 UI 요소와 겹쳐서 클릭이 어렵거나 가려지는 현상이 발생했습니다.
- **해결 방식:** `react-native-safe-area-context`의 `useSafeAreaInsets` 훅을 활용하여 기기의 안전 영역을 동적으로 측정했습니다. 하단 버튼 컨테이너(`ActionButtons.tsx`)와 주요 화면의 `ScrollView` 속성(`contentContainerStyle`)에 `paddingBottom: insets.bottom` 값을 추가하여, 기종에 상관없이 UI가 겹치지 않고 정상적으로 노출되도록 개선했습니다.

### 상호작용 결과 페이지 (`InteractionResultScreen.tsx`) 개선
- **문제점 1:** 분석 결과 요약 텍스트가 데이터가 없어서 분석하지 못한 경우에도 무조건 "상호작용이 감지된 항목이 있습니다"라고 출력되었습니다.
- **문제점 2:** 상호작용 리스트에서 '성분명 x 성분명' 형태로만 표시되어, 유저가 여러 약을 한 번에 검사할 때 "이 상호작용이 어떤 약 때문에 발생한 것인지" 직관적으로 파악하기 어려웠습니다.
- **해결 1 (조건부 텍스트 출력):** 데이터 상태를 구체적으로 분기 처리했습니다. 실제 상호작용이 있는 경우, 모든 데이터가 부족한 경우, 그리고 특이 상호작용이 없는 경우로 나누어 요약 메시지가 정확한 상황을 반영하도록 수정했습니다.
- **해결 2 (약품명 컨텍스트 추가):** 백엔드 응답(`InteractionItem` DTO)에 `drugName` 필드를 추가로 넘겨받아, 프론트엔드에서 상호작용 각 항목 위에 대상 약품명(`item.drugName`)을 작은 소제목 형태로 표시하도록 UI를 보완했습니다.
- **해결 3 (데이터 부족 UI):** 성분 데이터가 부족해 LLM이 분석을 못 한 항목은, 위험도 라벨 자리에 "데이터 부족"을 표시하고, 불필요하게 성분 곱하기(x) UI를 띄우지 않고 바로 안내 메시지를 볼 수 있도록 깔끔하게 정리했습니다.

### 상호작용 결과 페이지 - 아코디언 UI 적용
- **문제점:** 여러 약물을 동시에 분석할 경우 결과 내용이 너무 길어져 스크롤이 과도하게 길어지는 등 가독성이 크게 떨어졌습니다.
- **해결 방식:** 
  - 약품명(`drugName`)을 기준으로 상호작용 리스트를 그룹핑했습니다.
  - 리액트 네이티브의 `LayoutAnimation`을 사용하여 약품 이름 탭을 터치할 때마다 해당 약의 상호작용 상세 내용이 접히고 펴지는(Toggle) 아코디언 스타일 UI로 개편했습니다.
  - 접혀 있는 상태의 그룹 헤더에는 해당 약품 그룹의 등급 우선순위(**위험(WARNING) > 안전(SAFE) > 안내(그 외·레거시)**)에 맞는 라벨을 표시한다. (이후 등급 정책이 WARNING/SAFE 중심으로 바뀌면서, 헤더 로직도 같은 기준으로 맞춤 — 아래 §7 참고.)

## 2. 영양제 추천 기능 (Case B) 전면 개편

### 추천 로직 및 프롬프트 재설계
- **문제점:** 기존 영양제 추천 로직은 복용 중인 약과의 '금기 성분' 회피에 지나치게 초점을 맞추어 작동했습니다. 이로 인해 영양제 추천의 본질인 "기저 질환과 일상 컨디션 관리에 도움이 되는 영양소 추천"이라는 목적을 제대로 달성하지 못했습니다.
- **개선 내용:** 
  - 유저에게 '현재 증상이나 기저 질환(condition)'을 입력받도록 화면(`RecommendationScreen.tsx`)에 입력 필드를 추가하고, 백엔드 및 AI 서버로 해당 데이터를 전달했습니다.
  - AI 프롬프트를 전면 수정하여, 금기 성분에 대한 억지 비교는 제외하고, 대신 "복용 중인 약물과 상호작용(흡수 저하 등)이 없으면서, 입력한 기저 질환으로 인해 저하될 수 있는 컨디션 관리에 도움이 되는 성분"을 추천하도록 했습니다.
  - 의료법 준수를 위해 "치료", "예방" 등의 단어 대신 보수적 표현("활력 지원", "건강 유지")을 사용하도록 강력한 가드레일을 세웠고, 추천 영양소 개수도 최대 2개로 제한하여 정확도를 높였습니다.

### 추천 결과 페이지 신규 구현 (`RecommendationResultScreen.tsx`)
- AI가 분석한 **상호작용 및 안전성 검토 요약 내용**을 최상단 카드 형태로 배치했습니다.
- 추천된 영양소별로 한글 성분명, 영문 성분명, 추천 사유, 복용 주의사항을 보기 좋게 카드 형태로 노출했습니다.
- 추천된 영양소를 유저가 바로 구매할 수 있도록 쿠팡(Coupang)과 아이허브(iHerb) 검색 결과 링크 버튼(Linking.openURL)을 추가했습니다.

## 3. 백엔드 및 AI 연동 안정화

### 성분 데이터 누락 시 LLM 환각(Hallucination) 방지
- **문제점:** 약품 마스터 테이블(`drugs_master`)에는 약품이 존재하지만, 성분 테이블(`drug_ingredients`)에 매칭되는 데이터가 없는 경우, 빈 성분 데이터를 LLM에 던지게 되어 LLM이 임의로 약품 성분을 지어내서 답변하는 '환각' 현상이 발견되었습니다.
- **해결 방식:** 
  - 백엔드 서비스 레이어(`RecommendationService`, `InteractionCheckService`)에서 성분을 조회할 때, 일차적으로 `drug_ingredients`를 보고, 데이터가 없으면 이차적으로 `drug_permit_detail` 테이블에서 영문 성분명을 교차 검색하도록 로직을 보강했습니다.
  - 두 테이블 모두에서 성분 데이터를 찾을 수 없는 경우, **LLM 호출을 아예 생략**하고 백엔드 자체에서 "약품의 성분 데이터가 부족하여 분석을 진행할 수 없습니다"라는 정형화된 Fallback 응답 객체를 직접 생성하여 프론트로 반환하도록 수정했습니다. 이를 통해 AI가 거짓 정보를 생성하는 것을 원천 차단했습니다.

### 타임아웃 및 LLM 동시 호출 이슈 해결
- **API 타임아웃:** AI 모델의 응답이 15초를 넘어갈 때 프론트엔드 Axios에서 타임아웃 오류가 났습니다. 이를 해결하기 위해 Spring Boot의 `RestClient`는 Read 타임아웃을 90초로 연장하고, 프론트엔드의 Axios 호출도 `timeout: 120000`(120초)로 늘려주어 안정적으로 대용량 응답을 기다릴 수 있도록 처리했습니다.
- **빈 응답(Blank Response) 문제:** 
  - 터미널 로그를 확인한 결과, 유저가 3개 이상의 약품을 동시에 검색할 때, 백엔드에서 비동기 병렬 호출(`CompletableFuture`)로 여러 번의 API 요청을 동시에 Gemini로 쏟아부어 LLM 측에서 과부하가 걸려 빈 응답을 반환하는 문제가 확인되었습니다.
  - **해결 방안:** 백엔드의 `InteractionCheckService`에서 병렬 호출 로직을 걷어내고, 반복문(for-loop)을 통한 **순차 호출**로 변경했습니다. 또한, 각 LLM 호출 사이에 1.5초(`Thread.sleep(1500)`)의 의도적인 지연(Delay)을 두어 API Rate Limit 및 과부하 문제를 완화했습니다.

### 모델 ID·404 이슈 및 실제 사용 모델 (추가 정리)
- **상황:** `gemini-3-flash`, `gemini-3.0-flash` 등 이름으로 호출 시 Google GenAI API에서 `404 NOT_FOUND`(해당 `models/...` 없음)가 발생할 수 있음. 공식 **ListModels**로 확인한 결과, 키/엔드포인트마다 노출 목록이 다를 수 있으며, 본 프로젝트에서 확인된 호출 가능 ID에는 `gemini-3-flash-preview` 등이 포함됨.
- **조치:** `ai-server/app/services/gemini_service.py` 기본 모델을 **API 목록에 실제로 존재하는 ID**(예: `gemini-3-flash-preview`)로 맞춤. 잘못된 문자열로 404가 나지 않도록 함.
- **참고 스크립트:** `ai-server/scripts/list_gemini_models.py` — `.env`의 `GEMINI_API_KEY`로 `genai.Client` 후 `client.models.list()`를 돌려 **현재 키로 쓸 수 있는 모델 이름**을 터미널에 출력. 모델명을 바꿀 때는 이 스크립트로 먼저 확인하는 것을 권장.

## 5. Spring Boot — Hibernate SQL 로그 과다 출력 줄이기

- **문제:** 개발 중 백엔드 터미널에 Hibernate가 출력하는 SQL·바인딩 로그가 매우 길고 복잡해, 다른 로그 확인이 어려움.
- **설정 (`backend/src/main/resources/application.properties`):**
  - `spring.jpa.show-sql=false` 로 콘솔 SQL 출력 끔.
  - `logging.level.org.hibernate.SQL=WARN`, `logging.level.org.hibernate.type.descriptor.sql.BasicBinder=WARN` 등으로 Hibernate 관련 상세 로그 레벨을 낮춤.
- **적용:** 설정 변경 후 **Spring Boot 앱 재시작** 필요.

## 7. Case A 상호작용 — 등급 단순화 (WARNING / SAFE) 및 프론트 정리

### Python (`ai-server/app/api/v1/endpoints/interaction.py`)
- **정책:** LLM 출력 등급을 **WARNING(위험·병용 우려)** 와 **SAFE(문헌상 문제 없음)** 두 가지로 통일. **CAUTION·SYNERGY는 프롬프트에 길게 나열하지 않음** — 스키마가 두 값이면 충분.
- **후처리 (코드):**
  - `_normalize_level`: 모델이 옛 등급을 내면 WARNING/SAFE로만 보정 (예: CAUTION→WARNING, SYNERGY→SAFE 등).
  - `_normalize_and_limit_safe`: **WARNING은 개수 제한 없음**, **SAFE는 최대 2건**(앞쪽 우선).
- **예외:** Spring/직접 호출 시 성분이 비어 fallback 응답을 쓰는 분기는 기존처럼 `level=CAUTION` 등을 유지할 수 있음(프론트는 `nutrient === '알 수 없음'` 등으로 **데이터 부족** UI 처리).

### React Native (`InteractionResultScreen.tsx`)
- **CAUTION / SYNERGY 전용 색·라벨 분기 제거.**
- 헬퍼 **`normLevel`**, **`getLevelVisual`**: `WARNING`→위험(빨강), `SAFE`→안전(초록), 그 외(레거시 등급)→**안내**(회색).
- 그룹 헤더 배지: **WARNING > SAFE > 그 외(안내)** 순으로 요약.
- 카드 소제목: 영문 `WARNING`/`SAFE` 대신 **위험 / 안전 / 안내** 한글 라벨.
- `pnn-app/src/api/ocr.ts`: `InteractionItem.level` 주석을 `SAFE | WARNING` 중심으로 정리.

### 문서
- `docs/frontend-design.md`: 분석 결과 설명을 SAFE/WARNING 중심으로 한 줄 수정.
