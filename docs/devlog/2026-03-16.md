- **메인 주제**: 약품 상세 페이지 통합 데이터 조회 API (`GET /api/drugs/{drugId}`) 구현
- **배경**: 3월 14일 낱알식별/검색 결과(목록)를 구축했으므로, 목록에서 특정 약을 선택했을 때 보여줄 5개 탭(상단고정, 약품정보, 복약정보, 허가정보, DUR)의 상세 데이터를 내려주는 API가 필요함.

## 2. 완료한 작업 (Phase 2)

- 클라이언트(프론트엔드)가 각 탭을 클릭할 때 필요한 정보만 쏙쏙 빼서 쓸 수 있도록 가독성과 유지보수성을 극대화함.

### 2.2. 필요한 Repository(데이터 접근) 세팅
- 심평원 약가마스터 조회를 위한 `DrugPriceMasterRepository` 신규 생성
- 식약처 전문 허가정보 원문을 가져오는 `DrugPermitDetailRepository` 신규 생성
- 일반인 대상 쉬운 복약안내 정보를 가져오는 `DrugEasyInfoRepository` 신규 생성
- **트러블슈팅**: `DrugPriceMaster` 엔티티(Entity) 클래스 자체가 누락되어 있던 것을 발견하고 새로 추가함.

### 2.3. `DrugDetailService` 핵심 로직 구현 (데이터 조립기)
- `drugs_master` 테이블에서 `item_seq`(품목기준코드)를 가져와 중심축으로 삼음.
- 이 `item_seq`를 들고 4개의 테이블(허가정보, 복약정보, 약가마스터, 성분)을 돌며 필요한 데이터를 쓸어 담음.
- **DUR 검사 연동**: 약의 '영문 성분명'을 기반으로 `drug_contraindication`과 `dur_rules` 테이블을 뒤져서 병용금기, 임부금기 등의 경고 문자열 리스트를 추출함.

### 2.4. `DrugDetailController` 신규 생성 및 Swagger 연동
- 기존 검색용 컨트롤러(`DrugSearchController`)와 역할을 분리하기 위해 상세 전용 컨트롤러를 별도 생성함 (단일 책임 원칙 준수).
- API 엔드포인트: `GET /api/drugs/{drugId}` 추가 완료.

## 3. 학습 노트: Controller와 API 경로 문법 이해하기

오늘 마주쳤던 가장 중요한 개념인 **URL 경로 변수와 Swagger 인식 문제**를 정리해 둔다.

### 3.1. API 주소에 중괄호 `{ }`는 왜 쓰는 걸까? (Path Variable)
- **개념**: 스프링 부트에서 URL 경로의 특정 부분을 변수(파라미터)로 가로채기 위해 사용하는 문법이다.
- **예시**: `@GetMapping("/{drugId}")`
- **동작 방식**: 
  1. 클라이언트가 `GET /api/drugs/105` 라고 요청을 보낸다.
  2. 스프링 부트는 105라는 숫자를 `{drugId}` 자리에 들어온 동적인 '값'으로 인식하고 낚아챈다.
  3. 이 값을 메서드의 파라미터인 `@PathVariable("drugId") Long drugId` 에 자동으로 연결(주입)해 준다.
- **장점**: 수만 개의 약품마다 주소를 일일이 만들 필요 없이, 하나의 템플릿(공식)으로 모든 약품의 상세 조회를 처리할 수 있다.

### 3.2. Swagger UI에서 파라미터 입력 칸이 안 보이는 이유
- **원인**: Swagger가 코드를 읽어 들일 때, 스프링 부트의 어노테이션(예: `@PathVariable` 또는 `@RequestParam`)이 생략되어 있거나 명확하지 않으면 사용자가 입력할 파라미터가 없다고 오해해버린다.
- **해결책**:
  - `@Operation(summary = "...", description = "...")` 로 API의 목적을 확실히 적어준다.
  - `@Parameter(description = "약품 ID", required = true)` 로 파라미터의 존재와 필수 여부를 명시해 준다.
  - `@PathVariable("drugId")` 처럼 변수의 이름을 따옴표 안에 명확히 명시해 줘야 Swagger가 완벽하게 인식하고 화면에 입력 칸을 만들어 준다.

## 5. 세션 2: AI 서버 설정, Case A/B 검증, OCR·출력 정리 검토

### 5.1. API 키 및 환경 설정
- **문제**: API 키를 사전에 확인하지 않고 구현해, 호출 시 실패 발생
- **조치**: `ai-server/.env` 생성, `GEMINI_API_KEY` 설정 가이드 추가
- **config.py 개선**: 실행 위치와 무관하게 `ai-server/.env`를 로드하도록 `Path(__file__)` 기반 절대 경로 사용

### 5.2. Case A/B 직접 테스트
- AI 서버(8000) + Spring Boot(8080) 기동 후 Swagger/Invoke-RestMethod로 검증
- **Case A** (`POST /api/interaction/check`): drugId 20035 + 영양제 → 상호작용 등급 정상 반환
- **Case B** (`POST /api/recommendations/safe-nutrients`): drugIds [20035, 20036] → Probiotics, Vitamin D 추천 정상 반환

### 5.3. 테스트 데이터 정리
- **부정적 상호작용**: Warfarin+Vitamin K, Tetracycline+Calcium/Magnesium, Aspirin+Vitamin E 등 JSON 예시 + drugId 조회 SQL 제공
- **긍정적 상호작용(SYNERGY)**: 엽산+Methotrexate, 칼슘/비타민D+비스포스포네이트, CoQ10+스타틴, 프로바이오틱스+항생제, 비타민B12+메트포민
- **부하 테스트**: 성분 20~30개 이상 영양제 JSON 예시, 성분 많음 약품 조회 SQL 제공

### 5.4. Case A/B 설계 검토
- **다중 약품**: 실제 사용자는 1~5개 처방약을 동시 복용. 약 50개 성분 × 영양제 30~45개 성분 → LLM이 한 번에 처리하기 부담될 수 있음
- **규칙 기반 선필터(B)**: DUR은 약–약 상호작용용. 약–영양제 DB가 없어 실질적 효과 거의 없음
- **출력 단순화**: SAFE는 노이즈. CAUTION은 WARNING으로 통합 가능. **WARNING + SYNERGY만 반환**하는 방향으로 검토

### 5.5. OCR 모델 선택 및 검증
- **모델**: PaddleOCR-VL-1.5-GGUF (llama.cpp, llama-server)
- **테스트**: `app/test.py` — OpenAI 클라이언트로 `http://127.0.0.1:8081/v1` 호출, 이미지 base64 → OCR → 정규식 파싱으로 JSON 추출
- **포트**: Spring Boot 8080과 겹침 방지로 llama-server는 8081 사용

### 5.6. OCR 후 처리 흐름
- **결론**: OCR 출력 → 최소한의 파싱(JSON 구조화) → [선택] 한글→영문 정규화 → Case A API(drug + supplements) → Gemini로 상호작용 분석
- **파싱**: 필수. OCR 원문을 그대로 LLM에 넘기지 않음
- **정규화**: 약 성분이 영문이므로, 영양제 성분도 영문으로 맞추면 LLM이 더 안정적으로 판단

### 5.7. OCR 출력 형식 정리
- **프롬프트로 JSON 직접 출력**: PaddleOCR-VL은 문서 인식 특화 VLM이라, 지시 형식에 따라 JSON 출력이 안정적이지 않을 수 있음
- **현실적 선택**: 파싱(정규식) 유지 + 필요 시 프롬프트 개선 시도

### 5.8. 형식 정리용 중간 모델
- **옵션**: Gemini(이미 사용 중), Exaone(한국어), GPT-4o-mini, 로컬 소형 모델
- **추천**: 새 모델 도입 전에 Gemini로 OCR 텍스트 → JSON 정형화 시도. 한국어 비중이 크면 Exaone 검토
