# PNN 개발 세션 로그 — 2026-03-19

## 1. 세션 개요

- **메인 주제**: 영양제 성분표 촬영·분석 기능 완성, 이미지 업로드 안정화, UI 개선
- **배경**: 약X영양제 비교 페이지에서 영양제 사진을 촬영/선택 → OCR 분석 → 상호작용 비교까지의 전체 플로우를 프론트엔드·백엔드·AI 서버가 연동되어 동작하도록 구축함.
- **플랫폼**: 모바일(Expo Go) + 웹 모두 동작하도록 이중 경로 처리.

## 2. 영양제 이미지 업로드 안정화 (핵심 트러블슈팅)

### 2.1. 문제 상황

- **증상**: 프론트엔드에서 영양제 사진을 선택/촬영 후 백엔드로 전송 시 `MissingServletRequestPartException: Required part 'image' is not present` 발생
- **영향**: 이미지 분석 기능이 전혀 동작하지 않음

### 2.2. 원인 분석 (3가지 가설 검증)

| 가설 | 결과 | 설명 |
|------|------|------|
| **1. BASE_URL 문제** | ❌ 아님 | 약 검색/상세 API는 정상 동작 → 네트워크 연결·URL 설정 문제 없음 |
| **2. fetch + FormData 문제** | ✅ 일부 | React Native에서 `FormData`에 `{ uri, name, type }` 객체를 넣을 때 JS 브릿지·디버거 환경에서 boundary 누락 등 불안정 |
| **3. Spring Filter가 InputStream 소비** | ❌ 아님 | 커스텀 필터 없음, SecurityConfig만 기본 설정 |

**추가 발견**: `expo-file-system`의 `uploadAsync`는 **네이티브 레벨**에서 multipart 처리 → JS FormData 불안정성 우회 가능.

### 2.3. 적용한 해결책

#### (1) 모바일: `expo-file-system/legacy`의 `uploadAsync` 사용

- **설치**: `npx expo install expo-file-system`
- **주의**: SDK 55에서는 `uploadAsync`가 `expo-file-system/legacy`로 분리됨 → `import * as FileSystem from 'expo-file-system/legacy'`
- **에러**: `FileSystem.FileSystemUploadType.MULTIPART`가 `undefined` → 메인 패키지가 아닌 `legacy`에서 import 해야 함

```typescript
// ocr.ts (모바일 경로)
const uploadResult = await FileSystem.uploadAsync(url, imageUri, {
  httpMethod: 'POST',
  uploadType: FileSystem.FileSystemUploadType.MULTIPART,
  fieldName: 'image',
  mimeType: type,
  headers: { Accept: 'application/json' },
});
```

#### (2) 웹: `fetch` + `File` 객체로 FormData 구성

- **문제**: 브라우저 `FormData`는 `{ uri, name, type }`을 인식하지 못함 → `[object Object]` 문자열로 전송됨
- **해결**: `blob:` URL 또는 `data:` URL을 `fetch(uri)` → `blob` → `new File([blob], filename, { type })` → `formData.append('image', file)`로 실제 파일 전송

```typescript
// ocr.ts (웹 경로)
const imageResponse = await fetch(imageUri);
const blob = await imageResponse.blob();
const file = new File([blob], sanitizedFilename, { type });
formData.append('image', file);
```

#### (3) 플랫폼 분기

```typescript
if (Platform.OS === 'web') {
  return await uploadViaFetch(url, imageUri, type);
}
return await uploadViaFileSystem(url, imageUri, type);
```

### 2.4. 백엔드 관련 수정

| 항목 | 내용 |
|------|------|
| **NoClassDefFoundError (Publisher)** | `RestClient` + multipart 사용 시 `org.reactivestreams.Publisher` 필요 → `build.gradle`에 `implementation 'org.reactivestreams:reactivestreams:1.0.4'` 추가 |
| **ReadTimeoutException** | OCR + LLM 처리 시간이 길어 기본 타임아웃 초과 → `SupplementOcrService`에서 `SimpleClientHttpRequestFactory`로 read/connect timeout 각각 90초/10초 설정 |

## 3. 영양제 정보 요약 UI 개선

### 3.1. 제목 표시 변경

- **기존**: OCR/파일명 기반 (예: `b39dd7bf-7ce0-4b60-8fd4-4c3dbc2c22ff.jpeg`)
- **변경**: `분석완료된 영양제 1`, `분석완료된 영양제 2`, `분석완료된 영양제 3` (순서 기반)

### 3.2. 성분 표시 형식

- **기존**: 한 줄에 쉼표로 이어서 표시 (`비타민C, 비타민E, 오메가3`)
- **변경**: 
  - 리스트 형태로 각 성분을 줄바꿈하여 표시
  - **성분명 + 용량 + 단위** 모두 표시 (예: `• 비타민C 100 mg`)
  - `amount`/`unit`이 없으면 해당 부분 생략

### 3.3. 안내 문구 추가

- **문구**: `ai 모델이 인식 가능한 성분만 표시됩니다.`
- **위치**: 영양제 정보 요약 카드 하단
- **스타일**: 11px, 회색(`#9CA3AF`), `marginTop: 10`
- **표시 조건**: 영양제가 1개 이상이거나 분석 중일 때만 노출

## 4. 데이터 구조 및 비교 분석 플로우

### 4.1. 추출 데이터 (성분명, 용량, 단위)

- **파이썬 AI 서버** (`supplement_recognition.py`): PaddleOCR → Qwen 정제 → 정규식 파싱으로 `name`, `amount`, `unit` 3가지 추출
- **API 응답**: `{ name, nutrients: [{ name, amount, unit }] }`
- **프론트엔드**: `Supplement` 타입의 `ingredients`에 `Nutrient[]` (name, amount, unit) 저장
- **비교 분석 API**: `InteractionCheckRequestDto`의 `SupplementInput.NutrientInput`에 `name`, `amount`, `unit` 모두 전달됨 → **비교 분석은 이미 정상 동작**

### 4.2. 크롭 라이브러리 시도 및 롤백

- **시도**: `expo-dynamic-image-crop` — 자유 비율·코너 드래그 크롭 지원
- **문제**: 웹에서 미지원, 동작 불가
- **결과**: 롤백 → `expo-image-picker`의 `allowsEditing: true`, `aspect: [4, 3]` 유지

## 5. 파이썬 AI 서버 로깅 추가

### 5.1. 목적

- 이미지 분석 시 Paddle OCR 모델과 Qwen 모델의 각 단계 결과를 터미널에서 확인 가능하도록 함

### 5.2. 추가 위치

- **파일**: `ai-server/app/api/v1/endpoints/supplement_recognition.py`
