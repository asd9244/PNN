## 3. Case A 상호작용 검사 API 구현

### 3.1 엔드포인트

| 항목 | 내용 |
|------|------|
| Method | `POST` |
| Path | `/api/interaction/check` |
| Request | `InteractionRequestDto` (drugId, supplements) |
| Response | `InteractionResponseDto` (summary, source, interactions) |

### 3.2 DTO 구조

**요청 DTO (`InteractionRequestDto`)**

| 필드 | 타입 | 설명 |
|------|------|------|
| drugId | Long | drugs 테이블의 id |
| supplements | List\<SupplementInputDto\> | 복용 중인 영양제 목록 |

**공통 DTO (Case B에서도 재사용)**

| DTO | 필드 | 용도 |
|-----|------|------|
| SupplementInputDto | name, nutrients | 영양제 제품 1개 |
| NutrientInputDto | name, amount, unit | 영양제 성분 1개 (영문) |

**응답 DTO (`InteractionResponseDto`)**

| 필드 | 설명 |
|------|------|
| summary | 전체 결과 요약 |
| source | "RULE" (DB 규칙) / "LLM" (AI 분석) |
| interactions | InteractionItem 목록 |

**InteractionItem**

| 필드 | 설명 |
|------|------|
| nutrient | 영양제 성분명 |
| contraindicatedDrugIngredient | 부딪히는 처방약 성분명 |
| level | SAFE / CAUTION / WARNING / SYNERGY |
| description | 상호작용 설명 |
| actionGuide | 대처 가이드 |

### 3.3 설계 결정 — 변수명 통일

| 이전 | 이후 | 이유 |
|------|------|------|
| drugIngredient | contraindicatedDrugIngredient | 의미 명확화 |
| action | actionGuide | Python AI 서버 `interaction.py` 응답 필드와 동일하게 맞춤 |

- Python AI 서버 응답 DTO 필드명도 동일하게 수정하여 연동 시 일관성 확보

### 3.4 비즈니스 로직 (InteractionCheckService)

```
1. drugId로 Drug 조회 → 없으면 IllegalArgumentException
2. drug.main_ingr_eng를 "/"로 split → [영문 성분 목록]
3. (supplement × nutrient × drug_ingredient) 3중 루프
   → interactionRuleService.findByDrugIngredientAndNutrient(drugIngredient, nutrientName)
4. 매칭 규칙 있으면 → InteractionItem 생성 후 반환 (source="RULE")
5. 매칭 없으면 → TODO: Python AI 2차 분석 요청 (현재는 "알려진 상호작용 없음" 반환)
```

- `main_ingr_eng`가 null/빈 문자열인 drug는 로그 경고 후, 2차 AI 분석으로 넘길 예정 (TODO)

## 6. 데이터 테이블 정리 (3개 테이블 제거)

### 6.1 제거 배경

- **drug_easy_info**: e약은요 API. 상비약·OTC 위주, 약-약 경고 위주 → 처방약+영양제 use case와 부적합
- **ingredient_efficacy**: 심평원 성분약효. 성분 메타정보만, 경고문 아님 → RAG 컨텍스트로 활용도 낮음
- **knowledge_embeddings**: drug_easy_info + ingredient_efficacy 임베딩. RAG 소스 제거로 의미 없음

→ **drugs + drug_ingredients + interaction_rules** 중심 구조로 전환

### 6.2 제거한 테이블

| 테이블 | 이전 역할 |
|--------|-----------|
| drug_easy_info | e약은요 API. intrc_qesitm, atpn_qesitm 등 (약-약/약-음식 주의) |
| ingredient_efficacy | 심평원 성분약효. gnl_nm, div_nm 등 (성분 메타정보) |
| knowledge_embeddings | BGE-M3 임베딩. RAG 유사도 검색용 |

### 6.3 수행한 작업

| 구분 | 내용 |
|------|------|
| **스키마** | schema_relational.sql에서 ingredient_efficacy 제거, schema_vector.sql → DROP만 유지 |
| **DROP 스크립트** | `data/archive_setup/sql/drop_drug_easy_info_ingredient_efficacy_knowledge_embeddings.sql` 생성 |
| **Backend** | DrugEasyInfo, DrugEasyInfoRepository, DrugEasyInfoResponse 삭제, e약은요 배치/API 제거 |
| **ai-server** | vector_store.py 삭제, fetch_drug_easy_info/fetch_ingredient_efficacy 제거, interaction.py RAG 제거 |
| **설정** | application.properties drug-easy URL 제거, .cursor/rules 갱신 |
