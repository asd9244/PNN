## 2. ingredient_mapping 확장 시도와 발견

### 2.1 시도한 작업

drug_ingredients 테이블에 (mtral_nm, main_ingr_eng) 쌍이 있으므로, 이를 ingredient_mapping에 적재하면 매핑 수를 크게 늘릴 수 있다고 판단했다.

**생성한 SQL**: `data/scripts/sql/load_ingredient_mapping_from_drug_ingredients.sql`  
- drug_ingredients에서 DISTINCT (mtral_nm, main_ingr_eng) 추출  
- ingredient_mapping에 type='drug', source='drug_ingredients'로 INSERT  
- 적재 결과: 4,989건 (manual 165건과 합쳐 5,154건)

### 2.2 발견한 문제

터미널에서 ingredient_mapping 조회 결과를 확인한 뒤, **한글 1개 → 영문 N개** 매핑이 생기는 이유를 파악했다.

| value_kr (한글) | value_en (영문) |
|-----------------|-----------------|
| 구아이페네신 | Acetaminophen/Chlorpheniramine Maleate/**Guaifenesin**/Pseudoephedrine Hydrochloride |
| 구아이페네신 | Acetaminophen/Chlorpheniramine Maleate/Dextromethorphan/**Guaifenesin** |
| 구아이페네신 | (또 다른 약의 전체 성분 목록...) |

**원인**: 식약처 주성분 상세 API의 `MAIN_INGR_ENG`는 **해당 의약품의 전체 주성분 영문 목록**이지, 단일 성분의 영문명이 아니다. 같은 한글 성분(구아이페네신)이 여러 약에 들어가면, 약마다 전체 목록이 달라서 1:N 매핑이 된다.

**interaction_rules**는 `drug_ingredient = "Guaifenesin"`처럼 **단일 영문 성분명**으로 저장되어 있어, 전체 목록 문자열과는 매칭되지 않는다.

### 2.3 검토했던 대안들

| 제안 | 내용 | 결론 |
|------|------|------|
| "/"로 split해서 각각 저장 | 구아이페네신 → Acetaminophen, Chlorpheniramine, Guaifenesin, Pseudoephedrine | **불가** — 구아이페네신은 Guaifenesin만 해당. 나머지는 잘못된 매핑 |
| interaction_rules 복사 | Guaifenesin+Vitamin C 규칙을 Acetaminophen, Chlorpheniramine 등에도 복사 | **불가** — Acetaminophen+Vitamin C는 실제 상호작용이 없을 수 있어 잘못된 경고 |
| 각 성분에 규칙 추가 | Guaifenesin이 포함된 약의 모든 성분에 동일 규칙 추가 | **불가** — 동일한 이유 |

### 2.4 올바른 사용 방식

전체 목록을 split한 뒤, **각 토큰을 interaction_rules와 비교**하면 된다.

- `main_ingr_eng` = "Acetaminophen/Chlorpheniramine Maleate/Guaifenesin/Pseudoephedrine"
- split → ["Acetaminophen", "Chlorpheniramine Maleate", "Guaifenesin", "Pseudoephedrine"]
- 각 토큰으로 interaction_rules 조회
- "Guaifenesin"에 해당하는 규칙만 있으면 → 그 규칙만 표시
- Acetaminophen 등에는 규칙이 없으면 → 아무것도 표시하지 않음

즉, **데이터 구조는 문제 없고, 조회 로직만 split 후 비교**로 바꾸면 된다.

## 3. drugs.main_ingr_eng 추가 (중복 제거)

### 3.1 문제

drug_ingredients 테이블에서 같은 drug의 `main_ingr_eng`가 **row마다 동일하게 반복** 저장된다.

```
drug_id=1인 약 (성분 9개)
  row1: mtral_nm=포비돈,           main_ingr_eng="Mosapride Citrate Dihydrate"
  row2: mtral_nm=저치환도히드록시..., main_ingr_eng="Mosapride Citrate Dihydrate"
  ...
  row9: mtral_nm=스테아르산마그네슘, main_ingr_eng="Mosapride Citrate Dihydrate"
```

같은 문자열이 row 수만큼 중복 저장된다.

### 3.2 해결

`main_ingr_eng`를 **drug당 1건만** 저장하도록 drugs 테이블에 컬럼을 추가했다.

**변경 사항**:

| 파일 | 변경 내용 |
|------|-----------|
| `data/schema_relational.sql` | drugs에 `main_ingr_eng TEXT` 컬럼 추가 |
| `backend/.../domain/Drug.java` | `mainIngrEng` 필드 추가 |
| `backend/.../service/DrugDataService.java` | saveIngredientsPage 시 drugs.main_ingr_eng 업데이트 (null일 때만) |
| `data/scripts/sql/add_main_ingr_eng_to_drugs.sql` | 기존 데이터 이전용 ALTER + UPDATE |

**적재 로직**: drug_ingredients 저장 시, drug.main_ingr_eng가 비어 있으면 해당 값으로 채운 뒤 drug를 저장한다.

## 4. drugs.ingr_name vs main_ingr_eng 분석

### 4.1 질문

drugs.ingr_name에 `[M040044]농글리세린|[M040531]젤라틴|...` 형태로 전체 성분이 있으므로, 이를 활용해 한/영 매핑을 만들 수 있는지 검토했다.

### 4.2 검증 쿼리 결과

| item_name | ingr_cnt | eng_cnt |
|-----------|----------|---------|
| 가스프렌정(모사프리드시트르산염이수화물) | 9 | 1 |
| 발산정80밀리그램(발사르탄) | 6 | 1 |
| 가스리드정5mg | 10 | 1 |

**결론**:

| 소스 | 형식 | 예시 |
|------|------|------|
| **drugs.ingr_name** | 전체 원료 한글 (주성분+부형제) | [M040044]포비돈\|[M040531]젤라틴\|... (5~10개) |
| **drug_ingredients.main_ingr_eng** | 주성분 영문만 | Mosapride Citrate Dihydrate (1개) |

- ingr_name: 허가상세 API, **주성분+부형제** 전체
- main_ingr_eng: 주성분 상세 API, **주성분만**
- 개수·순서가 달라 **순서 매칭 불가**

### 4.3 데이터 갭

낱알식별/상세 UI에서는 **전체 성분 한글**을 보여주지만, 상호작용 검사에 쓸 수 있는 **영문 데이터**는 **주성분만** 있다. 부형제(포비돈, 산화티탄 등)의 영문 정보는 없다. 다만 상호작용 검사에는 주성분만 있어도 충분하다.

## 5. Case A/B 흐름 정리

### 5.1 Case A: 기복용 영양제 + 신규 처방약 충돌 검사

**순서1** → 사용자가 영양성분표 이미지 촬영 후 전송  
**순서2** → Spring Boot가 Python AI 서버로 OCR 요청  
**순서3** → OCR + LLM으로 nutrients JSON 정형화 (**성분명은 영문으로 출력**)  
**순서4** → Spring Boot가 user_supplements에 저장 (nutrients는 영문)  
**순서5** → 사용자가 신규 처방약 선택 → drug_id 확보  
**순서6** → drugs.main_ingr_eng 조회 → "/"로 split → [영문 성분 목록]  
**순서7** → Rule-based 1차 필터: (drug_ingredient, nutrient) × interaction_rules 매칭  
**순서8** → 1차에서 미결정이면 Python AI 서버로 RAG+LLM 심층 분석 요청  
**순서9** → 결과 반환 (SAFE / CAUTION / WARNING / SYNERGY 등)

### 5.2 Case B: 기복용 처방약 기반 안전 영양제 추천

**순서1** → 사용자가 기복용 처방약 리스트 확인 + 영양제 구매 의도 전달  
**순서2** → Spring Boot가 user_drugs에서 drug_id 목록 조회  
**순서3** → 각 drug_id로 drugs.main_ingr_eng 조회  
**순서4** → main_ingr_eng를 "/"로 split → [영문 성분 목록]  
**순서5** → 각 drug 성분에 대해 interaction_rules 조회 (findByDrugIngredient)  
**순서6** → CAUTION/WARNING인 nutrient 수집 → "피해야 할 성분" 목록 생성  
**순서7** → 경고 목록 + 처방약 정보를 Python AI 서버로 전달  
**순서8** → RAG+LLM으로 안전한 영양 성분 추천  
**순서9** → 추천 성분 + 쇼핑몰 딥링크 반환

### 5.3 nutrient 쪽 처리

**질문**: OCR 결과가 한글(비타민C, 마그네슘)이면 ingredient_mapping이 필요한가?

**결론**: LLM이 OCR→JSON 변환 시 **성분명을 영문으로 출력**하도록 하면, ingredient_mapping이 필요 없다. 영양제 성분명은 대부분 영문의 한글 표기(비타민C→Vitamin C)이므로, LLM이 영문으로 통일해 저장하면 된다. 외국 영양제도 같은 방식으로 처리 가능하다.

## 6. 설계 결정 사항

| 항목 | 결정 |
|------|------|
| nutrient 저장 형식 | LLM이 OCR→JSON 변환 시 **영문으로 출력** → ingredient_mapping 불필요 |
| drug 성분 조회 | drugs.main_ingr_eng split("/") → interaction_rules 직접 비교 |
| ingredient_mapping | Case A/B 흐름에서 **사용하지 않음** → 테이블 삭제 |
| 상세정보 없는 drug | main_ingr_eng 없으면(낱알 2.5만 vs 상세 3,800건) 약품명만 LLM에 전달 → 안내문 제공 |
| interaction_rules 미매칭 | 성분만 모아서 LLM으로 전달 (약 2.5만 drug vs 2천 성분 룰) |
| 복수 drug (Case B) | 처방은 대부분 복수 → 추후 별도 설계 |
| 표기 불일치 (Vitamin C vs Ascorbic acid) | 수작업 정제 불가, LLM 비용 부담 → 현실적 리스크 수용 |

## 7. ingredient_mapping 테이블 삭제

### 7.1 삭제 이유

Case A/B에서 drug는 drugs.main_ingr_eng split, nutrient는 LLM 영문 출력을 사용하므로 ingredient_mapping이 필요하지 않다.

### 7.2 수행한 작업

| 작업 | 내용 |
|------|------|
| schema_relational.sql | ingredient_mapping 테이블 정의 제거 |
| InteractionRuleService | IngredientMappingService 의존성 제거, 입력값 그대로 사용 |
| 삭제된 파일 | IngredientMappingService, IngredientMappingRepository, IngredientMapping, load_ingredient_mapping.py, load_ingredient_mapping_from_drug_ingredients.sql |
| 수정된 파일 | migrate_ingredient_mapping.sql, verify_interaction_rules.py |
| DB | DROP TABLE ingredient_mapping 실행 |

- `data/scripts/sql/drop_ingredient_mapping.sql` — 이후 재실행용
