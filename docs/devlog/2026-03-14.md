## 4. 학습 노트: QueryDSL 핵심 개념 이해하기

오늘 처음 도입한 QueryDSL의 복잡한 문법을 쉽게 이해하기 위해 핵심 개념 3가지를 비유와 함께 정리해 둔다. 향후 동적 쿼리 수정 시 이 개념들을 바탕으로 접근한다.

### 4.1. Q-Class (예: `QDrugsMaster`)
- **비유**: 식당의 **"정식 메뉴판"**
- **설명**: 기존 문자열 방식의 SQL(`"SELECT * FROM table WHERE item_namme = '...' "`)은 오타가 나도 실행해 보기 전까진 모른다는 치명적인 단점이 있다. 
- 반면 Q-Class는 엔티티를 바탕으로 만들어진 자바용 객체다. 개발자는 이 객체의 속성 자동완성을 이용해 쿼리를 짜므로, **오타로 인한 DB 에러를 컴파일 타임에 완벽하게 차단**할 수 있다.

### 4.2. BooleanBuilder
- **비유**: 검색 조건을 담는 **"장바구니"**
- **설명**: 사용자가 검색창에 입력한 조건의 개수는 매번 달라진다(동적 쿼리). `BooleanBuilder`는 처음에 텅 빈 장바구니 역할을 한다.
- 만약 사용자가 '제품명'을 쳤다면 `.and(이름 == 제품명)`을 담고, '색상'도 골랐다면 `.and(색상 == 빨강)`을 추가로 담는다. 이렇게 `if`문을 통과한 유효한 조건들만 차곡차곡 쌓아서 한 번에 DB에 전달하는 핵심 도구다.

### 4.3. JPAQuery / JPAQueryFactory
- **비유**: DB에 주문을 넣는 **"웨이터"**와 **"주문서"**
- **설명**: 
  - `JPAQueryFactory`(웨이터)를 통해 쿼리 작성을 시작한다.
  - `queryFactory.select(...).from(...).where(builder)` 처럼 영어 문장 쓰듯 **JPAQuery(주문서)**를 작성한다.
  - 주문서 작성이 끝나면 마지막에 **`.fetch()`** 메서드를 호출한다. 이 순간 쿼리가 데이터베이스로 날아가서 실제 데이터 결과값을 리스트 형태로 가져오게 된다.

### 4.4. 왜 굳이 `Custom` 인터페이스와 `Impl` 클래스로 쪼개는가? (Spring Data JPA의 딜레마 해결법)
- **기본 JPA의 한계**: `JpaRepository`를 상속받은 인터페이스에 `findByItemName`처럼 이름만 적어두면 스프링이 쿼리를 자동으로 만들어주는 기능은 매우 편리하다. 하지만 **조건이 사용자의 입력에 따라 동적으로 변하는 복잡한 검색**(예: 이름만 검색할 수도 있고, 모양+색상을 검색할 수도 있음)에서는 경우의 수만큼 메서드를 만들어야 하므로 불가능에 가깝다.
- **직접 구현의 딜레마**: 그렇다고 기본 인터페이스(`DrugsMasterRepository`)를 직접 클래스로 `implements` 해서 QueryDSL 코드를 짜려고 하면, 스프링이 공짜로 제공해주던 `findById`, `save` 같은 수십 개의 기본 메서드까지 전부 내 손으로 직접 구현해야 하는 대참사가 벌어진다.
- **해결책 (조립식 구조)**: 
  1. 내가 직접 짤 동적 쿼리만 **`Custom` 인터페이스**에 선언한다.
  2. 그 `Custom` 인터페이스만 **`Impl` 클래스**에서 100% 내 손으로 구현한다. (QueryDSL 사용)
  3. **메인 인터페이스**는 `JpaRepository`와 `Custom` 인터페이스를 **동시에 상속(extends)** 받는다.
- **스프링의 마법(동적 프록시)**: 앱이 켜질 때 스프링은 이 구조를 파악하고, **기본 기능은 자기가 만든 코드로, 커스텀 기능은 우리가 만든 `Impl` 객체로 알아서 연결(라우팅)해주는 가짜 객체(프록시)**를 덮어씌워 준다. 
- 결론적으로 개발자는 **기본 제공 기능과 복잡한 동적 쿼리를 하나의 Repository 인터페이스에서 모두 통일감 있게 호출**할 수 있게 된다.

### 4.5. IDE와 Q-Class 동기화 오류 (참고사항)
- **증상**: 간혹 `QDrugsMaster.java` 같은 생성된 클래스 파일명 탭에서 빨간색 에러 줄이 떴다가 막상 파일을 직접 열어보면 에러가 사라지는 현상이 반복된다.
- **원인**: 백그라운드에서 코드를 분석하는 IDE의 언어 서버(예: Eclipse JDT LS, 혹은 VS Code/Cursor의 자바 확장)가 새로 생성된 소스 코드 디렉터리(`build/generated/querydsl`)를 즉각적으로 동기화하지 못하고 캐싱된 옛 상태를 붙들고 있기 때문이다. 파일을 클릭해 여는 순간 언어 서버가 최신 상태로 강제 새로고침 되며 "아, 멀쩡한 파일이구나" 하고 에러 표시를 지우는 것이다.
- **해결**: 코드 구조 자체의 문제가 아니므로 무시해도 무방하다. 신경이 쓰인다면 **`Ctrl + Shift + P`** -> **`Java: Clean Java Language Server Workspace`** 명령을 실행하거나 IDE를 재시작하면 말끔히 해결된다.
